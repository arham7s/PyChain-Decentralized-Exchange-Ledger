Quick Start — 4 Steps
Step 01
🔑
Generate Wallets
Open the Wallets tab and click Generate Wallet. Each wallet gets a real ECDSA P-256 keypair generated in your browser — the private key never leaves your session. Create at least 2 wallets (sender + receiver). Each starts with ₹10,00,000 initial balance.
Step 02
📝
Sign a Transaction
Open the Transaction tab. Choose From/To wallets, pick an instrument (NIFTY, MCX Gold, NCDEX Wheat…), set side and amount, then click Sign & Add to Mempool. The transaction is cryptographically signed with the sender's private key before queuing.
Step 03
⛏
Mine the Block
All signed transactions sit in the Mempool. Set the difficulty slider (1–4 leading zeros), then click Mine Block. A Merkle root is computed for all pending transactions, then SHA-256 PoW runs until the hash target is met.
Step 04
📊
Inspect & Analyse
Click any block or ledger row to open the Inspector with live ECDSA signature verification. Hit Validate to verify every block's hash, prevHash linkage, and Merkle root. Scroll to Analytics to see all six live charts update.
Core Concepts
Cryptography
ECDSA Digital Signatures
Every transaction is signed by the sender's private key using ECDSA on P-256 — the same curve used in TLS and modern payment systems. The signature proves the sender authorised the transaction without revealing the private key. The Inspector verifies each signature on-chain using crypto.subtle.verify. A forged transaction shows SIG INVALID in red.
Blockchain
SHA-256 Proof-of-Work
Each block's hash is computed as SHA-256(merkleRoot + creatorId + timestamp + prevHash + nonce). The nonce increments until the hash starts with the required leading zeros. Difficulty 1 ≈ 16 tries, Difficulty 4 ≈ 65,536 tries. This is the same mechanism Bitcoin uses — higher difficulty = more tamper-resistant chain.
Integrity
Merkle Tree
Before mining, a Merkle root is computed from all transaction hashes in a binary tree structure. Pairs of hashes are combined and re-hashed until a single root remains. Tamper with even one character in one transaction and the Merkle root changes — making transaction-level fraud detectable without exposing all transactions.
Transaction Flow
Mempool
The mempool (memory pool) is a staging area for signed-but-unconfirmed transactions — exactly how Bitcoin Core and Ethereum nodes work. Transactions queue here until mined. In PyChain, Mine Block picks all pending mempool transactions at once. The mempool can be flushed to discard pending transactions without mining.
Smart Contracts
On-Chain Risk Rules
Three rules execute automatically. Balance Guard enforces solvency at mempool entry. Margin Call Trigger fires post-mining and injects a MARGIN_CALL alert for any wallet below the threshold — simulating automatic margin call logic like NSE, MCX, and NCDEX. Block Volume Alert flags concentration risk in the analytics dashboard.
Analytics
Six Live Charts
The Analytics section shows: cumulative settlement volume (line), instrument breakdown (donut), mining nonce effort (bar), block volume vs risk threshold (annotated bar), buy/sell/flow pressure (polar area), and post-settlement wallet balance distribution. All charts use Chart.js 4 with custom dark-mode styling and update the moment a block is mined.

At difficulty d, the hash must start with d zeros. SHA-256 output is pseudorandom, so the probability of any single hash meeting the target is 1 / 16^d. On average: Difficulty 1 → 16 tries, Difficulty 2 → 256 tries, Difficulty 3 → ~4,096 tries, Difficulty 4 → ~65,536 tries. Bitcoin continuously adjusts its difficulty so that network-wide, a block takes about 10 minutes regardless of total hash power. The Nonce chart in Analytics visualises this directly.
Real exchanges like NSE, MCX, and NCDEX monitor member positions continuously against margin deposits. If mark-to-market losses erode a member's margin below the maintenance margin, an automatic margin call is issued requiring top-up before the next settlement cycle — or positions are forcibly squared off. In PyChain, the Trigger simulates this: after each mined block updates all balances, any wallet below threshold automatically receives a MARGIN_CALL notification in the mempool. The ₹1L default threshold is intentionally modest for demonstration; in production this would be derived from SPAN/SPAN2 margin methodology — exactly the model you've built separately.
Three independent checks per block (skipping genesis): (1) recompute SHA-256(merkleRoot + creatorId + timestamp + prevHash + nonce) and confirm it matches stored hash; (2) confirm block[i].prevHash === block[i−1].hash, ensuring the chain is unbroken; (3) recompute the Merkle root from all transactions and confirm it matches stored merkleRoot. All three must pass for every block. If any fail, the specific block and failure type are reported.

In PyChain: address = "0x" + SHA-256(publicKeyHex).slice(0, 16). Ethereum uses a similar but different approach: keccak256(publicKey).slice(last 20 bytes). Bitcoin adds a RIPEMD-160 hash. The core principle is identical — the address is a deterministic one-way function of the public key. Two parties with the same private key always derive the same address, but an address cannot be reverse-engineered to reveal the private key. PyChain uses P-256 (secp256r1) rather than Bitcoin's secp256k1, but the security model is equivalent.
